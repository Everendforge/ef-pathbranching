# PathBranching Integration Architecture

This document defines how WorldNotion, PathBranching, and engine adapters should communicate while each tool can still be built and shipped independently.

## Product Flow

~~~text
WorldNotion
  canon vault, taxonomy, entities, links, validation
    |
    | reads stable IDs and metadata
    v
PathBranching
  narrative graph, Ink script references, conditions, consequences, export
    |
    | writes runtime package
    v
Engine adapter
  Unity first, then Godot, Unreal, and other runtimes
~~~

WorldNotion should not become the branching editor. PathBranching should not become the canon editor. Engine adapters should not require either authoring app at runtime.

This flow is a projection pipeline, not a copy pipeline. WorldNotion categories do not need to match PathBranching data classes, and PathBranching data classes do not need to match Unity or any other engine's runtime classes one-to-one.

## Communication Boundaries

### WorldNotion to PathBranching

WorldNotion provides a vault index and canon ontology.

PathBranching needs:

- vault root path
- entity stable IDs
- entity type
- entity display name
- aliases
- tags
- parent and child relationships
- custom taxonomy fields
- validation findings

PathBranching should treat the vault as read-mostly canon data. If PathBranching creates or updates canon entities later, it should do so through explicit user actions and preserve unknown frontmatter.

PathBranching should not assume that a WorldNotion type maps directly to an engine class. It should map WorldNotion entities into PathBranching data roles and projection classes first.

### PathBranching to WorldNotion

PathBranching can produce planning artifacts that WorldNotion may display.

Useful outputs:

- graph summary
- events that reference a canon entity
- unused canon references
- missing canon references
- narrative coverage reports
- generated scene or quest notes

This is optional for the first build. The first build only needs to read canon IDs from a vault.

### PathBranching to Engine Adapters

PathBranching exports runtime packages.

Engine adapters need:

- package metadata
- entry point
- canon references
- variables
- sequences
- branches
- events
- transitions
- conditions
- consequences
- script references
- engine-specific hints
- projection mappings
- validation report

The runtime package is the primary contract. Engine adapters should not read PathBranching private UI state.

### Engine Adapters to PathBranching

Engine adapters may report import and runtime findings.

Useful feedback:

- missing asset references
- missing Ink compiled files
- broken event transitions
- unsupported external functions
- engine-only runtime IDs
- playtest traces

This feedback can be imported as validation reports later.

## Shared Concepts

### Projection

A projection is a mapping from one layer to another.

Examples:

- WorldNotion `concept` -> PathBranching `KnowledgeEntry`.
- WorldNotion `scene` -> PathBranching `EventSource`.
- PathBranching `KnowledgeEntry` -> Unity knowledge data.
- PathBranching `EventNode` -> Unity event data.

Projections can be one-to-one, one-to-many, many-to-one, or skipped entirely.

### Canon Reference

A canon reference points to a WorldNotion entity by stable ID.

~~~json
{
  "id": "character:ridina",
  "kind": "character",
  "source": "vault"
}
~~~

For early compatibility, `id` may be either a namespaced ID such as `character:ridina` or a project-local legacy ID. Importers should preserve both when needed.

Canon references identify sources. They do not imply a target runtime class.

### Data Class

A data class is a PathBranching authoring concept that describes how selected data behaves in narrative logic.

Examples:

- `Speaker`
- `KnowledgeEntry`
- `SceneSetting`
- `EventSource`
- `InventoryItem`
- `UnlockCondition`

Data classes may be created from WorldNotion entities, authored directly in PathBranching, or generated by an importer.

### Graph Module

A graph module defines a visual node family, its ports, accepted data classes, and export behavior.

Examples:

- event node module
- knowledge condition module
- unlock consequence module
- dice check module
- Ink section module
- Unity `EventsData` projection module

### Sequence

A sequence is a route container, often tied to a playable character or campaign path.

In a Unity adapter, this may map to a sequence ScriptableObject.

### Branch

A branch is a named group of events.

In a Unity adapter, this may map to a branch ScriptableObject. Branches should be optional in the first PathBranching build.

### Event

An event is the main high-level graph node.

In a Unity adapter, this may map to event data. An event may reference an Ink source file and compiled Ink JSON file.

### Script Section

A script section is an internal Ink section, such as a knot or stitch.

The first build can parse and display script sections without owning full visual Ink editing.

### Transition

A transition connects one event or script section to another.

Transitions may be:

- explicit graph edges
- inferred from Ink diverts
- inferred from `SetNextEvent`
- engine-specific callbacks

The first build should store explicit graph transitions and also warn when Ink contains transitions that are not reflected in graph data.

### Condition

A condition gates a choice, state, event, or transition.

Examples:

- variable comparison
- canon knowledge unlocked
- item possession
- relationship threshold
- dice result threshold

### Consequence

A consequence mutates runtime state or emits an engine signal.

Examples:

- set variable
- unlock canon entry
- set next event
- add item
- modify relationship
- play timeline
- return to menu

## Runtime Package Layers

The package should have a stable base layer and optional adapter layers.

~~~text
Base Everend runtime data
  packageId, specVersion, entryNodeId, canonRefs, nodes, variables

PathBranching narrative data
  sequences, branches, events, transitions, scripts, externalFunctions, dataClasses, projectionRules

Engine adapter data
  unity, godot, unreal, custom engine hints
~~~

This keeps the package compatible with the current v0.1 schema while letting PathBranching express real engine adapter structures.

## Recommended Project Files

A PathBranching project should be file-based and portable.

~~~text
ProjectRoot/
  .everend/
    pathbranching.project.json
  Branching/
    graph.everend-branch.json
  Stories/
    *.ink
    *.json
  Exports/
    runtime-package.json
    validation-report.yaml
~~~

WorldNotion vaults can keep canon Markdown beside these files or in a sibling folder. The project manifest should store paths as relative paths whenever possible.

## First Data Contracts

PathBranching should define TypeScript domain models before UI work.

Initial models:

- `BranchingProject`
- `CanonRef`
- `DataClassDefinition`
- `ProjectionRule`
- `GraphModuleDefinition`
- `Sequence`
- `Branch`
- `EventNode`
- `ScriptRef`
- `Decision`
- `Outcome`
- `Condition`
- `Consequence`
- `Transition`
- `ExternalFunction`
- `EngineTarget`
- `ValidationFinding`

The same models should power:

- editor state
- file serialization
- runtime export
- validation
- Unity adapter import tests

## Build Order

1. Create PathBranching domain models and JSON examples.
2. Add data class and projection models before hardcoding Unity concepts.
3. Add a vault index adapter that can read WorldNotion-style entities.
4. Add a validator for broken event transitions, missing canon refs, and invalid projection inputs.
5. Add an Ink scanner that detects sections, tags, choices, diverts, and external functions.
6. Export a runtime package compatible with Everend Spec v0.1.
7. Build a Unity importer around projection rules for sequence data, event data, and Ink JSON references.
8. Build the event graph UI.
9. Add suite integration after the core packages are reusable.

## Tauri Strategy

Continue using Tauri, React, and TypeScript.

The important architectural rule is that app-specific UI should stay separate from shared domain packages.

WorldNotion and PathBranching can be finished as independent apps first if the shared logic is kept portable. Later, an Everend Forge suite shell can mount the same modules as DaVinci Resolve-style pages:

- World
- Branch
- Script
- Package
- Engine
- Playtest

This does not require changing away from Tauri. It requires discipline around package boundaries.

## Near-Term Decision

PathBranching should be developed as its own Tauri app, but its core should be built as if it will be embedded later.

That means:

- no business logic trapped inside React components
- no runtime export logic trapped inside Tauri commands
- no direct dependency on WorldNotion UI internals
- no Unity-only assumptions in core graph data
- engine adapters live outside the authoring core

This keeps the current approach and preserves the future suite path.
